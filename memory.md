# 好男人陀螺實驗室 - 多裁判即時賽事管理系統需求與架構說明書 (memory.md)

本文件詳實記錄了「好男人陀螺實驗室」在進行系統升級時的核心需求、技術架構決策與核心演算法邏輯，以利後續的系統維護、二次開發與需求對齊。

## 🏆 1. 核心業務需求與使用情境

本系統原為單機運作的陀螺分組與計分小工具，為因應現場賽事規模擴大，全面升級為**多裁判分布式大會實時評分與賽程同步系統**。

### 👥 現場使用情境 (User Story)
1.  **賽事主控端 (大會管理員)**：
    *   在賽事開始前，設定大會對戰台（裁判）數量與獲勝分數門檻（預設 4 分）。
    *   輸入或貼上參賽選手名單，可選擇一鍵隨機洗牌，並點擊生成對戰賽程。
    *   隨時可以在管理後台監管各個對戰台的即時對打選手與連線狀態，並可一鍵投射指定對戰台的 QR Code。
2.  **現場裁判端 (Referees)**：
    *   每位裁判分配到一個「對戰台編號」。裁判用手機掃描大會大螢幕或控制台提供的專屬 QR Code，快速登入該對戰台。
    *   登入後，手機計分板會進入「等待指派」遮罩狀態。
    *   一旦大會有比賽分派至該對戰台，計分板會自動解鎖並載入兩位選手的姓名。裁判只需專心計分（轉停 +1、擊飛 +2、爆裂 +2、極限 +3）。
    *   **無縫高速相機與慢動回放 (Slow-mo Replay)**：計分板上方整合了實時背相機預覽與一鍵錄製按鈕。裁判在每回合開始前點擊「開始錄影」，結束後點擊「停止」，**網頁會在一秒內自動以 4 倍慢速 (0.25x) 循環播放該回合影片**，並可手動微調 `0.25x`、`0.5x`、`1.0x`，幫裁判看清細微判定。當裁判**點擊任何計分按鈕時，回放自動關閉並重置回實時相機預覽**，無縫開啟下一回合錄影！
    *   裁判每次點擊得分，分數與得分歷史會**毫秒級即時同步**到大會總覽大螢幕。
    *   **左右位置交換功能 (Fighter Swap)**：裁判端保留「交換位置」按鈕。當實體現場對戰雙方互換對戰位置時，裁判點擊此按鈕，兩側選手的名稱、分數與對戰歷史會即時互換，並**即時同步更新至雲端與大會投影大螢幕**，確保大會看板的左右對戰視野與實體現場完全一致！
    *   **判決撤銷與復原功能 (Score Undo)**：新增「復原上一動」按鈕。當裁判因現場混亂、操作太快或手滑而「計錯分數」或「按錯得分對象」時，點擊該按鈕即可扣除上一次評分、遞減回合數，並**即時同步回滾雲端資料與大會大螢幕看板**。若該次誤觸已觸發比賽完賽（達到 4 分並鎖定），復原功能會**自動解鎖計分板、重啟對戰按鈕，並清除大螢幕完賽狀態與勝者**，完全免除重新開局的繁瑣動作，大大提昇大會容錯率！
    *   當某位選手達標（如 4 分）時，計分板鎖定並顯示「送出大會結果」按鈕，確認後送出，對戰台重回等待狀態。
3.  **大螢幕總覽看板 (Dashboard / Audience)**：
    *   在大會現場用投影機或大電視投放，以精美、高科技感霓虹電競風格呈現。
    *   上方以網格卡片即時同步呈現各個對戰台的對打選手姓名、即時分數與近期得分軌跡。
    *   下方呈現上下展開（手風琴式 Accordion）的淘汰賽樹狀圖。點擊每一輪的橫向標題可進行上下收闔與展開，展開時該輪的對戰場次以精美的霓虹網格水平平鋪排列，完賽後自動高亮晉級並動態點亮正在對戰中的場次與其分配的對戰台編號。

## 🏗️ 2. 技術架構決策

為確保本專案能**永久免費、零主機成本、極簡部署**，我們放棄了傳統的「PHP + MySQL + 輪詢」或「Node.js + WebSocket」伺服器架構，採用了 **無伺服器 (Serverless) 即時同步架構**：

```
       [ 裁判手機端 ]              [ 大會大螢幕看板 ]
     (Point/html.html)            (dashboard.html)
             \                           /
    (JS SDK)  \                         /  (JS SDK)
               v                       v
          [ ☁️ Google Firebase Realtime Database ]
                       ^
                       | (JS SDK)
                 [ 賽事管理後台 ]
                  (admin.html)
```

### 1. 靜態網頁代管：GitHub Pages
*   **優點**：100% 免費、超快 CDN 載入、直接對接 Git 版本控制。
*   **影響**：無法運行後端腳本（如 PHP、Node.js），整個系統的所有運算邏輯（賽程樹生成、晉級推導、場地分配）必須**完全在前端瀏覽器（Client-side JavaScript）中完成**，並直接向雲端資料庫寫入結果。

### 2. 即時資料庫：Google Firebase Realtime Database
*   **為什麼選擇它**：
    *   **真正即時 (WebSocket)**：裁判點擊得分時，雲端會即時發送 delta 更新，大螢幕同步延遲低於 0.1 秒，極大增強現場氣氛！
    *   **Serverless**：無須架設後端主機，全前端 SDK 載入，完美相容 GitHub Pages。
    *   **巨大免費額度**：1 GB 存儲與 100 個同時在線限制，對中小型實體賽事來說相當於**永久免費**。
*   **目前啟用的專案連線資訊**：
    *   專案 ID：`beybladex-c6fac`
    *   即時資料庫 URL：`https://beybladex-c6fac-default-rtdb.firebaseio.com`
    *   配置狀態：設定檔 [js/firebase-config.js](file:///Users/goodman/Desktop/個人資料/戰鬥陀螺/BeybladeX/js/firebase-config.js) 已成功配置為此真實專案金鑰，大會與裁判終端連線完全暢通。
*   **免登入本地模擬支援 (Developer Experience)**：
    *   考慮到用戶可能在無網路或尚未配置 Firebase Key 的情況下測試，我們在 `firebase-service.js` 中內建了 **LocalStorage 替代模擬機制**。
    *   若設定檔中的金鑰為預設佔位符，系統會自動在瀏覽器本地資料庫中進行數據儲存與事件派發，讓用戶在一台電腦打開多個分頁就能體驗完美的即時更新！

## 🧮 4. 核心演算法邏輯

### 1. 完美的單淘汰賽程樹與 BYE 分配演算法
當參賽人數 $N$ 不是 2 的次方時，會有 $P - N$ 位選手獲得首輪輪空（BYE）直接晉級（$P$ 為最接近大於等於 $N$ 的 2 的次方，如 $N=6 \rightarrow P=8$）。
*   **傳統做法的缺陷**：直接在名單尾部補 BYE，會造成 `BYE VS BYE` 對空打的無效比賽，導致有選手必須在第二輪才輪空。
*   **我們實現的完美分配演算法**：
    1.  首輪共有 $P/2$ 場比賽，輪空場次數為 $numByes = P - N$。
    2.  因為 $P - N$ 必定小於等於 $P/2$，我們將前 $numByes$ 場比賽全數排定為 `選手 VS BYE`。
    3.  剩餘的 $P/2 - numByes$ 場比賽排定為正常的 `選手 VS 選手`。
    4.  **結果**：首輪中，有輪空待遇的選手直接被設為 `"completed"` 並自動晉級至第二輪，其餘真實選手在第一輪正面對決。**徹底杜絕 BYE 互打與後續輪次輪空的瑕疵！**

### 2. 晉級關聯推導
對於 Round $R$ 的第 $M$ 場比賽：
*   其獲勝者將晉級到 Round $R+1$ 的第 $\lceil M/2 \rceil$ 場比賽。
*   若 $M$ 為奇數，晉級者填入下一場的 `player1`；若 $M$ 為偶數，填入 `player2`。
*   *代碼實作*：
    ```javascript
    const nextMatchId = `R${r + 1}-M${Math.ceil(m / 2)}`;
    const nextPosition = (m % 2 !== 0) ? "player1" : "player2";
    ```

### 3. 自動對戰台派發邏輯 (Stadium Auto-Allocation)
每當初始化賽事，或某場比賽完賽釋放對戰台時，系統自動執行以下指派：
1.  找出所有狀態為 `"idle"` 的對戰台。
2.  在 matches 中，篩選出狀態為 `"pending"`，且 `player1` 與 `player2` 皆已確定（不為 `null` 且不為 `"BYE"`）的待打比賽。
3.  依據**輪次優先 (Round 越早越優先)、場次優先 (MatchNum 越小越優先)**進行排序。
4.  將空閒對戰台與排在最前面的待打比賽進行雙向綁定更新，並將狀態設為 `"playing"`。

## 📱 5. QR Code 快速登入設計

為解決比賽現場裁判用手機登入時「手動輸入長網址極為不便」的痛點，系統提供了動態 QR Code 機制：
1.  **動態網址拼接**：
    系統在前端自動以 `window.location.origin + window.location.pathname` 動態取得當前的絕對部署路徑，並附加參數 `?stadium=stadiumId`。這使得不論在 `localhost` 區網 IP 還是 GitHub Pages，皆能產生 100% 精確的二維碼。
2.  **免套件超快生成**：
    我們利用免費開源的 `https://api.qrserver.com/v1/create-qr-code/` API，前端只需將拼接好的網址以 `encodeURIComponent` 帶入 `src`，即可完成免任何外部 JS 套件的超輕量 QR Code 載入。
3.  **雙入口便利性**：
    *   *管理端控制台*：點擊任何對戰台藥丸即可彈出霓虹 Modal 展示 QR Code。
    *   *裁判登入大廳*：裁判若在手機/平板上打開單機版評分頁面，可在最下方手動輸入對戰台編號，一鍵「直接登入」或「產生 QR Code」。

## 📹 6. 無縫高速相機與慢動作回放設計

為了解決賽事現場因毫秒差距（如停轉、爆裂先後）造成裁判判決爭議，系統特別實作了高科技的**本地即時慢動作回放系統**，兼具「極速流暢、不卡頓、零流量消耗」的特性：

### 1. 影像擷取與本地錄製
*   **MediaStream & Camera API**：調用手機背面高畫質主鏡頭（`facingMode: "environment"`），並鎖定在理想的輕量對戰解析度（如 640x480），以確保影片檔案極小，在手機記憶體中瞬間編碼完成。
*   **無聲錄製**：設定 `audio: false`，完全不錄音，除可避開各手機瀏覽器對麥克風權限的繁瑣審查，亦能進一步縮減影片 Blob 大小。
*   **MediaRecorder API**：優先選用瀏覽器相容性最佳的 `video/webm` 編碼器進行錄製；在不支援的設備（如部分 iOS Safari 舊版本）上，則自動向下相容 fallback 至 `video/mp4`。

### 2. 全螢幕判定與加大版自訂進度條 (Custom Scrubber)
*   **全螢幕慢動作裁判回放面板 (Modal Overlay)**：為了解決裁判在小螢幕（如手機）上觀看回放、不易點選進度條的痛點，系統設計了全新的滿版全螢幕 Modal Overlay 面板 `#replay-modal`。當錄影結束或點擊重播時會以全螢幕彈出，最大化視訊畫面。
*   **自訂加大進度條 (Custom Scrubber)**：使用加寬觸控點的 HTML5 滑動條（Scrubber），讓裁判可以用大拇指非常流暢且精準地左右拖拉進度，並能即時以毫秒單位 (`0.00s`) 顯示當前時間與總長度。
*   **影格微調 (Frame-by-Frame Adjust)**：控制面板整合了 `⏪ 逐格後退` 與 `⏩ 逐格前進` 按鈕，每點擊一次可精準前進或後退 `0.05 秒`（約 1-2 影格），對於多顆陀螺同時停止或飛出等爭議判定，提供完美的裁量精準度。
*   **原生全螢幕**：控制列提供「📺 全螢幕」按鈕，可直接請求 HTML5 原生 Video 全螢幕播放。

### 3. 多回合影片快取與歷程重播整合 (Round Preservation)
*   **非破壞性多回合快取**：全域物件 `roundReplays` 負責在網頁記憶體中快取整局對戰的每一回合錄影 Blob 數據，Key 為對應的對戰場次，絕對不被誤刪。
*   **歷程重播整合**：在「對戰歷程 BATTLE HISTORY」表格中最右側，為每一回合新增發光的 `🎥 重播` 按鈕。即使已經計分並同步大螢幕，裁判依然能隨時點擊歷程中任何一個場次的 `🎥` 按鈕，重新開啟全螢幕慢動作面板進行二次裁量與檢驗。
*   **自動釋放 (GC / Memory Management)**：僅在裁判點擊「重新開始」確認重置或大會「送出對戰結果」成功釋放對戰台時，系統才會安全地批次執行 `URL.revokeObjectURL()` 釋放快取，確保影片絕對不被誤刪且完全避免 Memory Leak。

### 4. HTML5 原生 Web Share API 手機媒體橋接 (Web Share API Mobile Photo/Media Bridge)
*   **一鍵喚醒系統原生面板**：當裁判在手機端（iOS Safari 或 Android Chrome）點擊「📥 下載/分享」時，系統會調用原生的 `navigator.share`，並將影片 Blob 檔案封裝成標準的 `File` 物件直接呼叫系統面板。
*   **直覺式本機儲存與播放**：這會**直接彈出手機作業系統原生的「分享與多媒體面板」**。在此系統面板中，裁判可以直接點擊系統提供的 **「儲存影片」**（即一鍵下載並存入本機相簿中），或者選擇 **「在其他媒體 App 或本機播放器中直接開啟觀看」**，與本機系統媒體功能完美對接！
*   **Fallback 下載降級**：若在不支援 Web Share API 的桌面端 PC，系統會自動降級至標準下載任務，直接將影片下載為 MP4。在不支援直接下載的舊 iOS Safari 設備上，則以新分頁開啟並提示長按儲存。

### 5. 一鍵計分與自動重置 (Seamless UX Reset)
*   **3-Click UX Workflow 依然完美運轉**：雖然加入了強大的全螢幕 Modal 面板，為防止裁判繁瑣的手動操作，當裁判點擊任何計分按鈕（如左側 +2 爆裂）時，核心計分函數 `addScore()` 除了呼叫 `closeReplay()` 之外，會**自動呼叫 `closeModalReplay()` 瞬間隱藏全螢幕回放面板**，將畫面無縫拉回實時相機預覽，省去手動關閉面板的動作，實現極速計分！

## 🛠️ 8. 近期核心重構與 UI/UX 升級紀錄 (2026 年 5 月)

為了提供大會與裁判端更完美的視覺饗宴與手機端操作手感，系統進行了以下重大重構與升級：

### 1. 手風琴式上下收闔賽事樹狀圖
- **上下堆疊佈局**：淘汰賽樹狀圖由原先過寬的「左右排列」重構為「上下垂直堆疊的手風琴結構」，一個區塊專注呈現一輪賽事。
- **智慧網格平鋪**：展開該輪時，所有的對戰組合卡片會以網格（Grid）水平平鋪排列。寬度不足時（例如手機）會響應式自動換行，徹底解決了原本看板左右寬度溢出的問題。
- **狀態持久化保留**：使用 `localStorage` 來記憶使用者的收闔與展開習慣。在 Firebase 毫秒級即時更新比分時，已收闔的輪次能維持穩定收闔，完全不會跳動或丟失狀態。

### 2. 單機分組工具 (TeamRandom) 深度現代化
- **風格統一與動態 Header 整合**：廢棄舊版 Bootstrap，全面引入 `modern.css` 並重寫 `style.css`，加入共享的 `header.html` 範本。利用 JS 智慧解析，在子目錄下動態加載時能自動校正相對路徑並正確高亮 active 狀態，實現全站設計一體化。
- **精美對戰卡片化**：將原本簡陋的分組列表重構為精美發光的對戰卡片（Player 1 Cyan 與 Player 2 Magenta 彩色雙向對立，帶 VS 徽章與霓虹 hover 發光），可點擊直達計分板。輪空選手以發光的青藍色 `seed-pill` 藥丸展示。

### 3. 手機尺寸導覽選單優化
- **響應式媒體查詢**：在 `modern.css` 底部新增了手機專屬 `@media (max-width: 768px)`。當寬度 $\le 768\text{px}$ 時，頂部標題自動縮小，`<header>` 轉為置中的垂直堆疊佈局。
- **流式對稱折行**：導覽按鈕 `<nav>` 寬度佔滿 100%，並設定能對稱地自動折行且放大了觸控熱區。徹底解決了手機螢幕按鈕折行擠壓與難以點選的問題，讓現場裁判以手機點擊時極為方便。
